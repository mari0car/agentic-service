# Core Concepts

## 1. The Paradigm Shift

Traditional backend development follows a pipeline:

```
Requirements (prose) → Code (compiled) → Binary (deployed) → Runtime (executed)
```

Every change cycles through the entire pipeline. The actual business intent exists only in step 1. Everything after is mechanical translation.

Agentic Service collapses this:

```
Specification (prose) → Runtime (interpreted by agent)
```

The specification *is* the runtime artifact. There is no intermediate code representation.

### Why This Works Now

Three capabilities converged:

1. **LLMs can reliably interpret structured natural language** into sequences of tool calls
2. **Tool-use protocols** (function calling, MCP) give LLMs deterministic interfaces to real systems
3. **Inference latency and cost** have reached levels acceptable for synchronous request handling in many use cases

The core insight: business logic is a function from inputs to tool-call sequences. LLMs are capable of computing that function from a prose description.

## 2. What Is an Agentic Service

An **Agentic Service** is a running process with three layers:

### Layer 1: Infrastructure (Code)

Implemented once in a programming language. Never changes per deployment.

- HTTP server accepting REST requests
- MCP server accepting tool invocations
- Built-in tools (database driver, HTTP client, filesystem, cache, messaging, crypto)
- Request routing and lifecycle management
- LLM provider integration
- Authentication and authorization enforcement
- Logging, metrics, health checks

This is the **runtime**. It is domain-agnostic. The same binary serves a task manager, an e-commerce platform, or an IoT data pipeline.

### Layer 2: Specification (Markdown)

Written by humans (or generated by other agents). Changes per deployment.

- Domain model definitions (entities, relationships, constraints)
- API route definitions (endpoints, methods, expected inputs/outputs)
- Business rules and workflows (validation, state machines, side effects)
- Data access patterns (which tables, which queries are expected)
- Error handling policies

This is the **business logic**. It lives in `specs/` as markdown files.

### Layer 3: Data (Database, Files, External)

The actual state the service manages.

- Database tables and rows
- Files on disk or object storage
- External service state
- Cache entries

### How They Interact

```
Request arrives (Layer 1)
  → Router identifies relevant spec files (Layer 1 reads Layer 2)
    → Agent receives: request context + spec content + available tools
      → Agent interprets spec, calls tools (Layer 1 executes against Layer 3)
        → Agent produces response
          → Runtime returns response to caller (Layer 1)
```

## 3. Specifications vs Code

| Aspect | Traditional Code | Agentic Service Spec |
|--------|-----------------|-------------------|
| Format | Programming language | Markdown prose |
| Execution | Compiled/interpreted by machine | Interpreted by LLM agent |
| Change cycle | Edit → build → test → deploy | Edit → done |
| Readability | Requires programming knowledge | Anyone can read |
| Precision | Exact (deterministic) | Approximate (probabilistic) |
| Performance | Nanoseconds per operation | Milliseconds to seconds per request |
| Expressiveness | Constrained by language syntax | Natural language (unconstrained) |
| Debugging | Stack traces, breakpoints | Agent reasoning traces |
| Testing | Unit tests, integration tests | Specification validation, behavioral tests |

### When This Trade-off Makes Sense

Agentic Service is appropriate when:

- Business logic changes frequently and redeployment is costly
- The logic is primarily data transformation, CRUD, and workflow orchestration
- Latency budgets allow 500ms-5s per request
- The team includes domain experts who are not programmers
- Rapid prototyping is more valuable than peak performance
- The service acts as a backend-for-frontend or internal microservice

Agentic Service is **not** appropriate when:

- Sub-millisecond latency is required (trading systems, game engines)
- The logic is compute-intensive (image processing, ML training)
- Determinism is legally required (financial calculations governed by regulation)
- The request volume exceeds what LLM inference can handle cost-effectively

## 4. The Agent's Role

The LLM agent inside Agentic Service is **not** a chatbot. It is a **function executor**.

For each incoming request, the agent:

1. Receives a structured prompt containing:
   - The request (method, path, headers, body)
   - The relevant specification file(s)
   - The available tools and their schemas
   - The system instructions for the runtime
2. Reasons about what tools to call and in what order
3. Calls tools (executes SQL, reads files, makes HTTP requests)
4. Produces a structured response (status code, headers, body)

The agent has **no memory between requests** (unless explicitly using a session/cache tool). Each request is an independent invocation. This is intentional: it makes the service stateless at the logic layer, just like traditional request handlers.

### Agent Constraints

The agent operates under strict constraints enforced by the runtime:

- **Tool whitelist**: can only use tools registered in the runtime
- **No arbitrary code execution**: cannot run shell commands or eval expressions
- **Scoped data access**: database connections, file paths, and external URLs are controlled by configuration
- **Time budget**: each request has a maximum processing time
- **Token budget**: each request has a maximum token consumption
- **Output schema**: must produce a valid HTTP response or MCP result

## 5. Specification Files as the Source of Truth

In a traditional service, the source of truth is code. In Agentic Service, the source of truth is the specification directory:

```
specs/
├── domain.md           # Entity definitions, relationships, constraints
├── api-routes.md       # Endpoint definitions and behavior
├── workflows.md        # Multi-step business processes
├── data-model.md       # Database schema and migration notes
├── error-handling.md   # Error policies and response formats
└── auth-policies.md    # Who can do what
```

These files are:

- **Version-controlled** (git)
- **Reviewed** via standard pull request workflows
- **Validated** by the runtime at startup (structural checks, not behavioral)
- **Hot-reloadable** (no restart needed for spec changes, optionally)

## 6. Comparison with Related Concepts

### vs Low-Code / No-Code Platforms

Low-code platforms replace code with visual builders (drag-and-drop, form-based configuration). They are still deterministic and constrained by the platform's abstraction. Agentic Service uses natural language, which is more expressive but less deterministic.

### vs Serverless Functions

Serverless functions (Lambda, Cloud Functions) still require code. They eliminate infrastructure management but not code authoring. Agentic Service eliminates both.

### vs API Gateways with Business Rules

API gateways with embedded rule engines (like Kong + custom plugins, or Apigee policies) handle routing and simple transformations. They cannot express complex multi-step business logic. Agentic Service handles arbitrary complexity.

### vs ChatGPT / AI Assistants

AI assistants are conversational agents for humans. Agentic Service is a machine-facing service that happens to use an LLM internally. It exposes REST/MCP APIs, not a chat interface. The LLM is an implementation detail, not the product.

### vs MCP Servers

MCP servers expose tools for agents to use. Agentic Service *is* an MCP server, but it also *uses* an agent internally. It sits at the intersection: it consumes tools (database, etc.) via an internal agent, and exposes tools (its API) to external agents.

## 7. Key Terms

| Term | Definition |
|------|-----------|
| **Agentic Service** | A backend service whose business logic is defined in prose and executed by an LLM agent at request time |
| **Specification (spec)** | A markdown file describing business logic, domain rules, or API behavior |
| **Tool** | A built-in capability the agent can invoke (e.g., `database.query`, `http.request`) |
| **Runtime** | The compiled binary that hosts the agent, tools, and API server |
| **Request Context** | The structured data passed to the agent for each request (method, path, body, headers, auth info) |
| **Agent Invocation** | A single agent execution handling one request, from prompt assembly to response generation |
| **Spec Directory** | The directory containing all specification files (`specs/`) |
| **Logic Interpretation** | The process by which the agent reads a spec and translates it into tool calls |
